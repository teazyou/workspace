# Window-Manager Code Review ‚Äî Dynamic Workflow

ultracode ‚Äî run a dynamic Workflow (multi-agent orchestration) that reviews and improves the window-manager config stack (AeroSpace + SketchyBar + JankyBorders + AutoRaise). First, if ./window-manager-code-review exists in the repo root, delete it, then create it fresh (with _spec/, reviews/, plans/, implementation/ subfolders) ‚Äî all artifacts go there. Drive these phases:

STUDY (step 1 of the Workflow ‚Äî ONE agent, Opus model at xhigh effort): the first step inside the single autonomous Workflow. It reads docs/window-manager/guide-window-manager.md ‚Äî and the relevant config folders/files as needed ‚Äî plus this workflow plan, then RETURNS (via structured output) the routing spec: the config-units (each: a key, a title, the exact files/globs it covers, and any notes) and the review lenses (key + label + focus), plus the hard constraints the downstream phases must respect. The Workflow script captures that return value and builds every later phase from it at runtime ‚Äî this is ordinary dynamic control flow, NOT a mid-run edit: the script awaits the study, then loops over the returned unit √ó lens lists to fan out. (The study also writes the same spec to ./window-manager-code-review/_spec/workflow-spec.json as a readable artifact.) The split STRATEGY is fixed and described in the phases below (one sub-agent per config-unit √ó lens; one plan per lens; sequential per-lens implement/verify) ‚Äî the study does NOT change that; it only supplies the unit and lens lists, read from the guide/code rather than hardcoded. The lists typically land on ~5 units (e.g. aerospace-core, aerospace-scripts, sketchybar-core, sketchybar-plugins, borders-autoraise) and 3 lenses (bugs/correctness, performance, clean-code/refactor ‚Äî templating, dedup, shared libs, named constants), but the study reads them from the current guide/code.

The remaining phases run inside that SAME Workflow, fanned out from the study's returned lists:

REVIEW (parallel ‚Äî one sub-agent per config-unit √ó lens from the spec): each reviewer reads the REAL files for its unit, cites file:line + evidence, respects the spec's hard constraints (Bash 3.2 compat, symlink source-of-truth model, NO macOS notifications, manual linting only), and writes window-manager-code-review/reviews/<lens>--<unit>.md.

PLAN (parallel ‚Äî one agent per lens): consolidate that lens's findings into an ordered, concrete change plan ‚Üí window-manager-code-review/plans/<lens>-plan.md. Plan only; no edits.

COHERENCE (BLOCKING all later steps ‚Äî single agent): reconcile the per-lens plans, resolve every cross-lens conflict, drop redundant/unsafe changes, emit one globally-ordered changeset partitioned by lens ‚Üí window-manager-code-review/coherence-review.md.

IMPLEMENT then VERIFY (sequential, in this order): implement bugs, implement performance, implement clean-code, then verify/fix bugs, verify/fix performance, verify/fix clean-code. Implementers make minimal live edits to the real (symlinked) config files, re-reading each file right before editing, run `bash -n` on every edited shell script, and write window-manager-code-review/implementation/<lens>-implementation.md + <lens>-verification.md. Do NOT shellcheck/lint, NOT reload aerospace/sketchybar/borders, NOT git commit or push.

REPORT: read all artifacts + `git diff`, write window-manager-code-review/REPORT.md with an exec summary, per-lens found‚Üíplanned‚Üíimplemented‚Üíverdict, a table of every file changed, deferred/skipped items, and manual reload follow-up (the `aerospace-restart` alias).
